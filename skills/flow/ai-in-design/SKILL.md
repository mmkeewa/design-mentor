# ai-in-design

> ⚠️ **LIVING SKILL.** Область сдвигается быстро (модели, инструменты, паттерны). Сверяйся с актуальными гайдлайнами Google PAIR, Apple ML HIG и текущим состоянием инструментов перед выдачей рекомендаций. То, что работало в 2024, может быть не оптимально в 2026.

Два разных вопроса, часто путают:
1. **Как проектировать продукт, в котором есть AI.** Паттерны, доверие, ошибки.
2. **Как использовать AI в процессе дизайна.** Скорость, качество, границы.

Head отвечает за оба и держит их отдельно.

## Часть 1 — дизайн AI-продуктов

### Ключевые паттерны AI-интерфейсов

**Input paradigms.**
- **Prompt bar** — открытое поле, пользователь пишет что хочет. Свобода + барьер входа.
- **Structured input** — dropdown / form / preset. Меньше свободы, ниже барьер.
- **Multi-modal** — text + voice + image. Богаче, но сложнее в UI.
- **Conversational** — chat-like. Хорошо для итеративных задач, плохо для one-shot.

**Output paradigms.**
- **Streaming** — токены появляются один за другим. Даёт ощущение скорости, снижает perceived latency.
- **Structured response** — таблица / карточка / список. Читаемее, чем сплошной текст.
- **Multiple options** — 3–5 вариантов, пользователь выбирает. Хорошо для creative задач.
- **Refinement** — «Продолжить», «Сократить», «Перефразировать» — итеративная работа с output.

**Trust patterns.**
- **Provenance** — «Ответ на основе...». Ссылки на источники.
- **Confidence indicators** — «Модель уверена на X%». Осторожно: пользователи плохо интерпретируют вероятности.
- **Uncertainty language** — «Возможно», «Обычно», «Не уверен» — калиброванная неуверенность.
- **Undo / editable** — всегда можно отменить или отредактировать.
- **Show your work** — «Как я подумал». Стало стандартом с reasoning-моделями.

**Error handling.**
- **Graceful fallback** — модель не смогла → предлагаем альтернативный путь (форма, поиск, поддержка).
- **Retry with variation** — «Попробовать иначе», не тот же промпт снова.
- **Human escalation** — эскалация к живому человеку, если критично.

### Doverие как дизайн

**Правило калиброванной уверенности.** Модель не должна звучать увереннее, чем она есть. Плохо: «Правильный ответ — 42». Хорошо: «Скорее всего, 42, но проверьте вручную для важных решений».

**Прозрачность про AI.** Пользователь должен знать, что говорит с AI, а не с человеком. Особенно критично в customer service.

**Отзывчивость к обратной связи.** Thumbs up/down на каждом ответе. Если пользователь исправил ответ — учись от него.

**Границы AI.** Явно скажи, где AI не работает. «Не даю финансовых рекомендаций», «Не консультирую по медицине». Лучше явно, чем неточно.

### Human-in-the-loop (HITL)

Не «AI делает всё автоматически». HITL — паттерн, где человек включён в критических точках.

**Когда HITL обязателен:**
- Финансовые операции (утверждение платежа, KYC).
- Медицинские / юридические рекомендации.
- Действия с необратимыми последствиями (удаление, отправка).
- Критические business decisions.

**Level of automation:**
1. AI предлагает → человек делает.
2. AI делает → человек утверждает.
3. AI делает → человек может отменить.
4. AI делает автономно (только для reversible / low-stakes).

Head должен явно выбрать уровень для каждой AI-фичи и обосновать.

### Evals — метрики AI-фич

**Не «точность модели»**, а **пользовательские metrics**:
- **Success rate** — % задач, решённых без переделки.
- **Time-to-answer** — от запроса до полезного ответа.
- **Iterations to satisfaction** — сколько refinement-циклов до финала.
- **Trust score** — «доверяете ли ответу?» — периодически.
- **Escalation rate** — % задач, переданных в human support.
- **Regeneration rate** — как часто просят «попробовать ещё».

Каждая AI-фича — свои evals **до релиза**. Иначе не поймёшь, работает ли.

### Uncertainty UI — как показывать

Плохие способы:
- Просто число «73%» — пользователи не читают вероятность.
- Только цвет (жёлтый = не уверен) — цвет unreliable без context.

Хорошие способы:
- **Категориальные шкалы:** «Уверен» / «Скорее да» / «Не уверен».
- **Список альтернатив:** «Возможные ответы: A (вероятнее), B, C».
- **Совет с оговоркой:** «Скорее всего X. Проверьте, если решение важное».
- **Просим больше контекста:** «Мне нужно уточнение: [конкретный вопрос]».

## Часть 2 — AI в процессе дизайна

### Где AI ускоряет

- **Ideation** — веер вариантов концепций за минуты. Не финал, а топливо для [creativity-ideation](../creativity-ideation/SKILL.md).
- **Копирайтинг черновиков** — first draft для empty states, ошибок, notifications. Финал редактируешь.
- **Прототипы кода** — vibecoding даёт живой прототип быстрее Figma-макета. Проверить flow быстрее.
- **Синтез research'a** — 12 интервью → кластеры → инсайты. Сжимает часы работы.
- **Аудит** — прогнать чек-лист по экранам (a11y, editorial policy, consistency).
- **Документация** — черновик component doc / spec для handoff.

### Где AI **не** заменяет человека

- **Продуктовое суждение** — что важно, что нет, куда идти. Head-level decision.
- **Понимание пользователя** — глубинные интервью человек ведёт лучше.
- **Стратегические ставки** — AI не знает контекст компании, домен, стейкхолдеров.
- **Финальный визуальный крафт** — AI даёт черновики, финал — руки человека.
- **Ответственность.** Если AI-фича сломалась — отвечает Head, не модель.

### Vibecoding

Практика: описываешь UI словами → AI генерит рабочий код (React / Vue / SwiftUI) → правишь итеративно. Работает через Claude Code, v0, Cursor, Aider.

**Плюсы:**
- Живой прототип быстрее макета.
- Можно проверить logic + UX сразу.
- Handoff в разработку — часть кода уже готова.

**Минусы:**
- AI генерит код, дизайнер не всегда понимает что.
- Без design-system'ы получаешь inconsistent UI.
- Пропущенные состояния (empty / error / loading) — AI забывает.

**Правило:** vibecoding + жёсткая дизайн-система + чек-лист состояний = рабочая практика. Без системы = быстрый мусор.

### Skills — что это, зачем

Skill — упакованная инструкция для AI-агента (Claude). Формат: `.md` файл с frontmatter (когда активировать, что делает) + тело (метод, шаблоны, ссылки).

Head-уровень: пишешь свои skills под повторяющиеся задачи — audit, research synthesis, moodboard, brief-parsing. Скилы шарятся с командой, versionируются в git.

Пример скилов: [design-discovery](https://github.com/mmkeewa/design-discovery), [design-mentor](https://github.com/mmkeewa/design-mentor) (этот).

### MCP (Model Context Protocol)

Открытый протокол для подключения AI-агентов к внешним источникам данных и инструментам. Figma MCP, GitHub MCP, Notion MCP, Slack MCP — расширения возможностей Claude/GPT.

Для дизайнера:
- **Figma MCP** — Claude может читать / писать в Figma-файл, генерировать компоненты по спеке.
- **GitHub MCP** — Claude ведёт код-репозиторий (например, design-system как код).
- **Custom MCP** — свой инструмент под свой процесс.

Спецификация: [modelcontextprotocol.io](https://modelcontextprotocol.io/).

### Промпт-паттерны для дизайнера

**Explicit role.** «Ты senior product-designer с 15-летним опытом в fintech. Твоя задача — [X]».

**Structured output.** «Ответь в формате: 1) Диагноз (1 предложение), 2) 3 варианта решения, 3) Что я забыл спросить».

**Few-shot examples.** «Пример хорошего результата: [X]. Пример плохого: [Y]. Мой запрос: [Z]».

**Constraints.** «Ограничения: только 1 экран, без иконок, для пользователей 65+».

**Iterative refinement.** «Ответ хороший, но сделай ещё короче и убери жаргон».

**Reasoning aloud.** «Разложи вслух, как ты пришёл к этому. Потом дай финальный ответ».

### Границы

Head должен явно **сформулировать команде**, где AI используется, где нет.

**Обычно AI-friendly:**
- Первичный черновик копирайтинга (с ревью).
- Ideation-варианты.
- Прототипы для дискуссии.
- Синтез research'a (с проверкой).
- Аудит по чек-листу.

**Обычно AI-restricted:**
- Клиентские тексты в проде без правки человеком.
- Персонализация без объяснимости.
- Автоматические действия с деньгами / данными пользователя без HITL.

## Ключевые концепции

**AI — тул, не цель.** «Внедрим AI» — не задача. «Ускорим activation на 20%» — задача, AI может быть инструментом.

**Trust > accuracy.** Точность модели важна, но **воспринимаемое доверие** — часто важнее для продукта. Плохо откалиброванный «уверенный» ответ хуже честного «не знаю».

**Uncertainty is a feature, not a bug.** Хорошо показанная неуверенность = доверие. Скрытая = разочарование при ошибке.

**Human-in-the-loop is not weakness.** Автоматизация ради автоматизации приводит к катастрофам. Head выбирает уровень automation осознанно.

**Skills = replicable process.** Одна и та же задача, одинаковый результат, разные проекты. AI без скилов = неповторяемая работа.

**Domain-специфика.**
- **Fintech.** HITL обязателен для транзакций. Uncertainty UI критичен для советов. Explainability для регуляторки.
- **E-com.** AI-рекомендации — базовый паттерн. Ошибки в рекомендациях = потеря продажи, не катастрофа.
- **Edtech.** AI-tutor: важна калибровка «правильно/неправильно», нельзя запутать ученика.

1. **Похвала.** «Хорошо, что задумываешься про паттерны, а не только про модель» *(или)* «Хорошо, что видишь границы AI, а не тащишь везде».
2. **Диагноз.** «У тебя AI-фича без uncertainty UI — доверие сломается на первой ошибке» *(или)* «Vibecoding без дизайн-системы — быстрый мусор».
3. **Паттерн / инструмент.** Streaming / HITL / skill — конкретно.
4. **Evals или границы.** Как поймёшь, что работает / где не используешь.
5. **Следующий шаг.** «Спроси про [product-metrics] для настройки evals, или про [editorial-policy] для тона AI-ответов».

## Ловушки

- **AI как маркетинг.** Внедрили ради «у нас AI». Пользователю без разницы, ему нужна задача решена.
- **Модель врёт с уверенностью.** Плохая калибровка → пользователь принимает решение → доверие сломано.
- **Автоматизация без HITL.** AI отправил платёж не тому → регуляторный кошмар.
- **Vibecoding без системы.** Быстро, но inconsistent UI, состояния забыты.
- **Скилы как разовый скрипт.** Написали один раз, не поддерживаем, за 3 месяца устарел.
- **Разработчики + PMы делают AI-фичу без дизайна.** Prompt bar + streaming response — вроде работает, но паттерны promotable ниже среднего.
- **Игнор границ.** Команда пишет копирайт AI'ом и заливает в прод без ревью → в поддержке жалобы на странные ошибки.
